# Build System Standard

Reference for the build toolchain used across repos in this org (MSW, pypulsepal, rpi-cam-ensemble, …).
The canonical template is **[larsrollik/templatepy](https://github.com/larsrollik/templatepy)** — use it for new repos and as the reference for what the standard looks like.
This doc covers: how to use the template, what the template does not handle (Zenodo, vendoring, README structure), and how to retrofit existing repos.

---

## Using the copier template

### New repo

```sh
copier copy gh:larsrollik/templatepy my-new-project \
  --data "project_name=My Project" \
  --data "project_slug=my_project" \
  --data "project_description=Short description" \
  --data "author_name=Lars B. Rollik" \
  --data "author_email=l.b.rollik@protonmail.com" \
  --data "github_username=larsrollik" \
  --data "github_repo=my-project" \
  --data "year=2026"
cd my-new-project
git init && git add -A && git commit -m "chore: initial commit from templatepy"
uv sync --extra dev
uv run pre-commit install --hook-type pre-commit --hook-type commit-msg
```

Then:
- Create the GitHub repo and push
- Set up PyPI trusted publishing (pypi.org → Publishing → Add pending publisher: `release.yml`)
- Enable branch protection: require `CI` status check; allow `github-actions[bot]` to bypass

### Retrofit existing repo (external/ repos, pypulsepal, etc.)

Copier can overlay a template onto an existing repo. It creates `.copier-answers.yml` so future `copier update` works.

```sh
# 1. Preview what would change (safe — writes nothing)
copier copy gh:larsrollik/templatepy /tmp/preview-output --data ...

# 2. Compare with the existing repo manually
diff -rq /tmp/preview-output existing-repo/ --exclude='.git'

# 3. Apply selectively — copy only the files you want to adopt
#    Typical targets for external/ repos:
cp /tmp/preview-output/.github/workflows/ci.yml      existing-repo/.github/workflows/
cp /tmp/preview-output/.github/workflows/release.yml existing-repo/.github/workflows/
cp /tmp/preview-output/.pre-commit-config.yaml     existing-repo/
cp /tmp/preview-output/VERSION                     existing-repo/  # if missing
cp /tmp/preview-output/CITATION.cff                existing-repo/  # fill in metadata

# 4. Align pyproject.toml manually — don't overwrite, merge:
#    - commitizen: version_provider="commitizen", version_files=["VERSION"]
#    - hatch.version: source="vcs", fallback-version matching VERSION
#    - optional-dependencies: move from dependency-groups if present
```

**Note:** Do not blindly overwrite `pyproject.toml`, `README.md`, or `src/` — those have existing content. Workflow files, `.pre-commit-config.yaml`, and `VERSION` are safe to replace wholesale.

### Receiving template updates

In any repo generated by copier (has `.copier-answers.yml`):

```sh
copier update
```

Copier fetches the latest templatepy, replays your answers, and shows a diff. Review and commit the result.

## Zenodo integration

**No GitHub Actions step needed.** The webhook fires when `release.yml` creates a GitHub release.

### One-time setup per repo

1. [zenodo.org/account/settings/github](https://zenodo.org/account/settings/github) → flip the toggle ON for the repo
2. Add `CITATION.cff` at the project root (see template below)
3. Push the first versioned tag — Zenodo archives it and creates a record
4. Copy the concept DOI from the Zenodo record; add it to `CITATION.cff` and the README badge

### CITATION.cff template

```yaml
cff-version: 1.2.0
message: "If you use this software, please cite it as below."
type: software
title: "PackageName: short description"
repository-code: https://github.com/larsrollik/repo
license: GPL-3.0
authors:
  - family-names: Rollik
    given-names: Lars B.
    orcid: https://orcid.org/0000-0003-0160-6971
identifiers:
  - type: doi
    value: 10.5281/zenodo.XXXXXXX
    description: Zenodo concept DOI (resolves to latest version)
```

Use concept DOI (not version DOI) — it resolves to the latest version and is stable across releases.

### README badge

```markdown
[![DOI](https://zenodo.org/badge/doi/10.5281/zenodo.XXXXXXX.svg)](https://doi.org/10.5281/zenodo.XXXXXXX)
```

Note: old badge format (`zenodo.org/badge/{github-repo-id}.svg`) is broken since Zenodo migrated to InvenioRDM. Always use the `doi/` path form.

## README structure

Required sections in order:

```
# PackageName
badges (DOI, PyPI)
One-line description

## Example usage
## Installation
## Citation          ← only if Zenodo DOI exists
## License & sources
## Useful references  ← optional
```

**Badges:**
```markdown
[![DOI](https://zenodo.org/badge/doi/10.5281/zenodo.XXXXXXX.svg)](https://doi.org/10.5281/zenodo.XXXXXXX)
[![PyPI](https://img.shields.io/pypi/v/my-package.svg)](https://pypi.org/project/my-package)
```

Omit DOI badge until Zenodo is set up. Do not use a Black badge or a Ruff badge. Do not hardcode version strings in README body.

### License & sources section

Required in every repo. States the project license and attributes all vendored code.

```markdown
## License & sources

This software is released under the **[LICENSE-NAME](LICENSE)**.

`_vendored_file.py` is vendored from [upstream-project] vX.Y.Z ([LICENSE-TYPE], Copyright © YEAR Author).
The original source is at [upstream URL].
```

## Vendoring third-party code

### File header

Every vendored file must start with:

```python
# Vendored from: <upstream project name> (<upstream URL>)
# Version: <upstream version or commit hash>
# License: <SPDX identifier>  Copyright (c) <year> <copyright holder>
# Modifications: <"none" or one-line summary>
```

### Exclude from ruff line-length

```toml
[tool.ruff.lint.per-file-ignores]
"src/my_package/_vendored_file.py" = ["E501"]
```

### License compatibility

| Project license | Can vendor |
|---|---|
| MIT / BSD / Apache-2.0 | MIT, BSD, Apache-2.0, PSF |
| GPL-3.0 | MIT, BSD, Apache-2.0, LGPL, GPL |
| Proprietary | only permissive (MIT, BSD, Apache-2.0) with explicit permission |

MIT and BSD-2/3 are safe to vendor into anything. GPL code cannot be vendored into a non-GPL project.

## Contributor and authorship guidelines

> No single standard adopted for this org yet. Record here when a decision is made.

- **ICMJE** — academic authorship: requires substantial contribution + drafting + approval + accountability. Written for papers; "data acquisition" maps loosely to "writing code."
- **CRediT** — 14 contributor roles (Software, Validation, Visualization, …). CITATION.cff supports CRediT natively. Finer-grained than ICMJE; allows crediting non-author contributors.
- **all-contributors** — GitHub bot that credits any contribution type (code, docs, ideas, …). High overhead; suited to large OSS projects.
- Collect ORCIDs at contribution time — chasing them before a release is unreliable.
- ORCID format in CITATION.cff: full URL `https://orcid.org/0000-0000-0000-0000`.

## Legacy repo upgrade checklist

For repos that already exist and can't be generated fresh. After applying: run `copier update` once `.copier-answers.yml` is added.

- [ ] Replace `.github/workflows/` with templatepy versions (ci.yml, release.yml, docs.yml)
- [ ] Add/update `.pre-commit-config.yaml` with templatepy version
- [ ] `pyproject.toml`: switch build backend to `hatchling` + `hatch-vcs`; add `[tool.hatch.version] source="vcs"`; align `[tool.commitizen]` to `version_provider="commitizen"`, `version_files=["VERSION"]`; switch `[dependency-groups]` → `[project.optional-dependencies]`
- [ ] Add `VERSION` file at project root; content must match `[tool.commitizen] version`
- [ ] Move package to `src/<package_name>/` if still flat layout
- [ ] Add `CITATION.cff` and enable Zenodo webhook
- [ ] Add DOI badge to README after first release
- [ ] Set up PyPI trusted publishing on pypi.org
- [ ] Run `pre-commit install --hook-type pre-commit --hook-type commit-msg`
- [ ] Vendored files: add header, add `README.md` attribution, add ruff `per-file-ignores`

## Known toolchain gotchas

### mypy v2: `import-untyped` vs `import`

mypy v2 splits missing-stubs errors into two codes:

- `import` — no stub and no `py.typed` marker (was the only code in mypy v1)
- `import-untyped` — the package has no inline types and no stubs package available on PyPI

`ignore_missing_imports = true` suppresses `import` but **not** `import-untyped`. Packages without a stubs package require an explicit ignore:

```toml
# pyproject.toml — suppress both codes cleanly
[[tool.mypy.overrides]]
module = "yaml"
ignore_missing_imports = true
```

Or install the stubs package (preferred when one exists):

```toml
# pyproject.toml — dev extras
dev = [
    "types-PyYAML",   # stubs for pyyaml; silences mypy import-untyped
    ...
]
```

Always prefer the stubs package over a per-module override — it gives real type information.
